# 目标

按照 `docs/牌桌体验优化规划-2到4.md` 依次实施第 2、3、4 项，并从第 2 项开始：

1. 为每位玩家增加半身人物表现，支持待机、出牌、过牌、出完等状态。
2. 将当前玩家固定显示在牌桌最下方，其余玩家按真实座次顺时针旋转。
3. 改造结算等待界面，允许返回牌桌或退出房间，同时保持下一局确认同步。

# 当前上下文

- 前端使用 React、TypeScript 和 CSS，玩家座位由 `PlayerSeat` 与 `pos-1`～`pos-6` 控制。
- 后端使用 Go 和 WebSocket，房间状态、最近动作、下一局确认均由服务端广播。
- 已有十二生肖头像，可作为半身人物设计的身份基础。
- 已有开局荷官发牌动画、倒计时、最近动作、语音、背景音乐和结算确认名单。
- 本轮首先实施第 2 项，但完整目标包括规划中的第 2～4 项。

# 默认假设

- 半身人物采用轻量化网页动画，优先保证六人同时在线时流畅。
- 第一版人物沿用十二生肖形象，不引入真人角色，避免与现有头像系统割裂。
- 不改变真实座位号、队伍、出牌顺序和牌局规则。
- 结算返回仅关闭遮罩，不等于退出或取消确认。
- 退出已结束房间需要服务端明确支持，避免被“对局中禁止退出”拦截。

# 风险

- 六个持续动画可能影响低性能手机，需要提供动画开关并尊重减少动态效果设置。
- 座位视觉映射可能导致队友、倒计时、最近动作指向错误，必须以真实座位号验证。
- 结算遮罩关闭后仍需保留重新打开确认名单的入口。
- 玩家退出可能破坏房主转移、确认人数和自动开局逻辑。
- 新素材可能增加首屏体积，需要压缩并延迟加载。

# 执行方案

1. 建立人物动画状态模型，将最近动作映射到对应真实玩家。
2. 制作并接入轻量生肖半身人物素材与待机、出牌、过牌、出完动画。
3. 完成人物动画设置开关、移动端适配和性能保护。
4. 建立“真实座位 → 当前玩家视角位置”的统一映射函数。
5. 将玩家卡片、人物动画、倒计时等全部接入同一映射。
6. 完成六个自我座位视角的自动化验证。
7. 将结算遮罩改为可关闭面板，并提供重新打开入口。
8. 增加结束状态下安全退出、确认同步和房主转移处理。
9. 完成六人确认、有人退出、断线重连和自动下一局测试。
10. 做最终桌面端、移动端、构建、测试、安全和回滚审查。

# 验证方式

- 前端类型检查与生产构建通过。
- Go 单元测试与新增状态测试通过。
- 本地 Docker 前后端健康检查通过。
- 浏览器实际检查六个座位、人物动画、设置开关和结算返回流程。
- 使用 WebSocket 自动化脚本验证确认、退出和自动开局。
- 检查控制台无错误，关键素材全部返回成功。

# 回滚方案

- 每个阶段保留可独立回退的代码改动。
- 不删除现有生肖头像、原座位号或结算数据结构。
- 若人物动画影响性能，可通过设置开关关闭，基础玩家卡片仍可使用。
- 若新结算交互异常，可恢复为原模态窗口，同时保留后端确认状态。
